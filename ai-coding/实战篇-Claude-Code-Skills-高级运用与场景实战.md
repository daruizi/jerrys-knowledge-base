# 🚀 实战篇：基于 Claude Code Skills 构建强大的自动化模式

> 本文参考了《Claude Skills 完全构建指南》，将带你从理论走向实践。
> 在开始前，我们先明确一个核心理念：**Skills 并不生产工具，Skills 负责编排工具与传授专业知识。**

把它想象成去五金店找装修师傅：
- **问题优先**：你带了一个问题（“我想装新橱柜”），Skill 会按照正确的顺序使用不同的工具（电钻、锤子）来解决它。
- **工具优先**：你买了一把新电钻，Skill 是教你如何最规范地使用这把电钻的手册。

下面，我们将通过 **5 个经典的实战模式**，教你一步步搭建高能的工作流。

## 前置必读：Skill 的“启动引擎” YAML

在开始写模式之前，所有的 `SKILL.md` 顶部必须包含一段 YAML 配置（Frontmatter）。这是决定 Claude 会不会调用你这个 Skill 的生死开关。

**保底必写模板：**
```yaml
---
name: your-skill-name
description: [触发短语] + [核心功能] 
---
```
> **💡 排坑警告：**
> 1. `name` 和文件夹名字必须一模一样（只用 kebab-case，如 `code-review`）。
> 2. `description` 必须详细！坏的写法：“审查代码”。好的写法：“查阅代码中潜在的安全漏洞，当用户询问审查代码、review、检查安全时使用。”
> 3. **绝对禁止**在 YAML 块内出现 XML 标签（`< >`），这会触发安全拦截！

**高阶可选配置（可选）：**
除了必填项，你还可以通过以下高阶字段强化控制：
```yaml
allowed-tools: "Bash(python:*) Bash(npm:*) WebFetch" # 权限控制：限制它只能用哪些工具
metadata: # 任意自定义键值对，有助于归档与管理
  author: YourName
  version: 1.0.0
  mcp-server: projecthub # 标明它依赖哪个 MCP
```

## 架构师思维：设计 Skill 的三大核心法则

在你动手写下这 5 种模式之前，请在心中牢记高质量 SOP 的三个灵魂：

1. **递进式披露 (Progressive Disclosure)**：绝不要把几万字的规范直接全贴在 `SKILL.md` 里，平时只加载最核心的 YAML 触发器，被触发后才加载 `SKILL.md`，遇到不懂的参数再让它去通过工具读取额外的 `references/xxx.md`。
2. **可组合性 (Composability)**：你的 Skill 不是宇宙的中心。Claude 可以同时加载多个 Skills，你的设定应该像一个乐高积木，能跟其他自动化的流程相互协作，不要假设自己是唯一可用的能力。
3. **可移植性 (Portability)**：写好一份标准的 Skill，它在客户端、 Claude Code CLI 以及官方 API 下的运行逻辑完全相同，一套 SOP 三端通用。

## 模式一：单向业务流水线 (顺序工作流程编排)

**🎯 适用场景**：必须严格按照 1、2、3、4 步骤执行的操作，绝对不能乱套。
**🛠️ 实战案例**：一套“员工/客户入职分配系统”。

**实战配置（`SKILL.md`）：**

```markdown
---
name: onboard-customer
description: 引导新客户完成从开户到迎新邮件的完整入职流程。当用户要求“办理新客户入职”或“onboard”时使用。
---

# 新客户入职流程指导

当执行新客户入职时，请严格按照以下步骤顺序操作：

### 第一步：创建账户
1. 收集用户的 name、email 等基础信息。
2. 使用适当的终端命令或数据写入工具在数据库/文件记录客户档案。

### 第二步：支付配置
1. 生成支付初始化链接。
2. 必须等待第一步成功后，才能将系统状态标记为 `Pending Payment`。

### 第三步：下发迎新通知
1. 读取 `assets/welcome_email_template.md` 模板。
2. 将模板中的信息替换为客户的实际档案信息。
3. 提示我信息已准备好发送。
```
**核心心法**：不要只告诉它终点在哪，要像对实习生一样，列出每一步及依赖关系。

## 模式二：跨界联动大师 (多 MCP 协调)

**🎯 适用场景**：你需要让 Figma、云盘、Slack、任务看板等多个互不相连的系统联动起来。
**🛠️ 实战案例**：设计稿转开发流程。

**实战配置（`SKILL.md`）：**

```markdown
---
name: design-handoff
description: 将设计稿转化为开发任务。当用户提供一个设计文件并要求“移交开发”、“建排期”或“通知组员”时使用。
---

# 设计图移交开发流水线

你是一个跨部门协调专家。在处理此类任务时，请协调以下系统进入工作：

### 阶段 1：解析与暂存 (Drive)
1. 观察当前项目目录下的设计资源（assets）。
2. 在本地 `docs/design_specs/` 中生成开发排版说明书。

### 阶段 2：任务创建 (Linear/Jira 模拟)
1. 阅读说明书，拆解出至少 3 个具体的前端任务。
2. 如果连接了任务 MCP 服务器，请调用之批量建卡。如果未连接，请在终端输出一段可以直接复制的 Markdown 任务列表。

### 阶段 3：全员通知 (Slack 模拟)
1. 任务建好后，整理一份简短的移交通知。
2. 输出包含“所有资产链接”和“对应任务编号”的通知文案，以便我直接发到群里。
```
**核心心法**：这是发挥 MCP 服务潜力的核心舞台，将数据当作接力棒在不同阶段中传递。

## 模式三：自我打磨机制 (迭代精炼)

**🎯 适用场景**：初稿往往很烂，需要反复校对校验的场景（写长篇文章、重构祖传代码）。
**🛠️ 实战案例**：代码生成与语法校验闭环。

**实战配置（`SKILL.md`）：**

```markdown
---
name: strict-refactor
description: 对复杂前端组件执行重构并自我校验质量。当用户要求复杂重构或优化组件时使用。
---

# 严格重构机制

为了保证代码不被改崩，你需要在输出最终结果前，进行自我迭代：

### 1. 产出初稿
- 在本地的 `/tmp/` 或 `.temp_refactor/` 目录下生成你修改后的组件草稿。

### 2. 质量审查
运用你的命令行工具执行以下操作进行检查：
1. 运行 `npm run lint:js` 检查草稿文件。
2. 阅读控制台的报错信息。
3. 如果有报错：诸如“未使用变量”、“类型错误”等。

### 3. 精炼循环 (Refinement Loop)
如果上一步发现了报错：
- 根据错误信息修改刚才的草稿。
- 再次执行审查，直到控制台没有任何报错。

### 4. 最终定稿
只有完全通过校验，才允许替换掉原有的源代码文件。
```
**核心心法**：赋予 AI “停下来跑个测试”的权力，让它自己纠错，而不是改完直接丢给你一个烂摊子。

## 模式四：智能分拣中心 (上下文感知工具选择)

**🎯 适用场景**：根据上传的素材格式不一，智能采取不同的处理方案。
**🛠️ 实战案例**：自动归档助手。

**实战配置（`SKILL.md`）：**

```markdown
---
name: smart-storage
description: 自动智能整理并分类项目中的文件。当用户说“整理桌面”、“归档这些文件”时使用。
---

# 智能文件归档决策树，请按照本策略执行：

### 分拣判定逻辑：
请先检查我要归档的文件类型与大小，并执行不同的操作：

- **大型媒体文件 (`>5MB` 的图片/视频)**：必须移动到 `assets/media/` 并输出“已归档离线大文件”。
- **研发代码与脚本 (`.js`, `.py` 等)**：必须移动到 `src/scripts/`，如果没有找到对应目录则创建它。
- **纯文本和笔记 (`.txt`, `.md`)**：必须移动到 `docs/` 并按年月建立子文件夹存放。
- **配置与杂项**：留在原地。

### 透明度要求
归档结束后，向我输出一张表格，告诉我你刚刚移动了哪些文件，以及你是基于上述哪条规则移动它们的。
```
**核心心法**：这就像给 Claude 脑子里植入了一个 `if-else` 分支表，省得每次都强调“这个放哪，那个放哪”。

## 模式五：带着“家规”干活 (领域特定智能)

**🎯 适用场景**：你所在的行业有极其严格的红线（如金融合规、医疗隐私等），这是最高优级别的审查。
**🛠️ 实战案例**：数据脱敏后置处理器。

**实战配置（`SKILL.md`）：**

```markdown
---
name: process-logs
description: 提取并分析用户的日志文件。当用户提供线上日志让你分析时使用。
---

# 合规日志分析师

你正在分析线上真实数据的日志，**在进行任何计算或工具调用之前，请优先进行脱敏合规审查**：

### 1. 触发前置合规安检 (Compliance Check)
读取提供的日志文件内容，执行以下强制替换策略：
- 如果遇到手机号（连续的 11 位数字）：自动打码为 `138****0000`。
- 如果遇到邮箱（包含 `@` 的字符串）：自动隐藏前缀，如 `***@gmail.com`。
- 如果遇到任何明文的 `password=` 或 `token=`：立刻删除整行。

### 2. 安全数据处理
只有进行完脱敏后，你才能在一份**临时文件**上执行用户的分析指令。

### 3. 生成审计记录
工作完成后，附带说明：“本次日志检索触发了合规检查，屏蔽了 X 处手机号和 Y 处敏感 Token。”
```
**核心心法**：行动前先讲合规。将具体的专业领域知识和合规法则直接“缝合”到了业务逻辑前面。

## 进阶必备：Skill 的标准工程级目录结构

当你的 SOP 非常复杂时，把所有内容全塞进一个 `SKILL.md` 会导致 Claude 上下文爆炸。官方推荐**递进式披露**的标准目录结构：

```text
your-skill-name/
├── SKILL.md                  # 必填项：主控文件，只写核心步骤和高频示例
├── scripts/                  # 选填项：脚本库（如 .py, .sh 测试脚本）
├── references/               # 选填项：长篇参考文档（API 手册、长篇规范）
└── assets/                   # 选填项：模板文件（如 Excel 模板、Markdown 模板）
```

**💡 核心避坑：**
- **绝不要**在 Skill 文件夹里放 `README.md`！所有的说明交给 `SKILL.md` 和 `references/`。README 应该放在项目的外面供人类阅读。
- 在 `SKILL.md` 中，随时可以这样指挥：“*遇到 API 报错，请查阅 `references/api-guide.md`*”。Claude 会在需要时才去读取它，这就叫“递进式披露”。

## 如何测试与打磨你的 Skill？

写好 Skill 后，如何知道它真的好用？官方强烈推荐 **三维测试法** 和使用 **skill-creator**。

### 1. 官方神器：使用 `skill-creator`
在 Claude.ai 插件目录（或 Claude Code）中，有一个官方自己写的 Skill 叫 `skill-creator`。
遇到写不出 `description` 时，直接呼叫它：`“用 skill-creator 帮我写一个处理项目初始化的 Skill”`，它会帮你自动生成标准格式、避开触发陷阱。

### 2. 三维测试法
- **触发测试（最关键）**：测试它该跳出来时跳出来，不该跳时不跳。如果你发现 Claude 动不动就乱用这个 Skill（**过度触发**），赶紧在 description 里加上“负面触发词”（如：不适用于纯 Python 代码任务）；如果你喊半天不出来（**触发不足**），就在 description 补充触发关键词。
- **功能测试**：跑一遍全流程，故意给它错的数据，看它能不能捕捉错误并在内部消化（参考模式三的自我打磨）。
- **性能对比**：原来你要人工对话 15 轮，消耗 12000 tokens。用了 Skill 后变成了 2 轮对话，消耗 6000 tokens。这就是成功的 SOP 构建。

### 3. 如何定义这套 Skill 是成功的？（成功指标）
优秀的架构师需要拿数据说话。你可以用以下标准来衡量：
- **量化指标**：在 90% 的相关提问上精准触发（不需你手动 /调用）；在期望的工具调用次数内完成工作，0 API 报错。
- **定性指标**：你不再需要像哄小孩一样告诉 Claude “下一步该怎么做”，它能一次性跑完全程不需要人类中途纠错。

## 急诊室：常见错误与排错指南

在开发技能时遇到阻碍是常态，收好这份避坑清单：

- **报错 `Could not find SKILL.md`**：去检查文件名是否完全拼对大小写（不能是 `skill.md` ），绝不能包含多余空格。
- **报错 `Invalid frontmatter`**：大部分是因为 YAML 格式少了三个破折号 `---` 闭合，或者是引号没有成对出现。
- **Skill 变得极度卡顿 / 乱回答**：通常是因为 `SKILL.md` 正文超过了 5000 字，或者当前目录启用的 Skills 数量太多。这时候必须执行**“递进式披露”**，把长篇大论转移到只读文件 `references/` 里，减轻 Claude 脑部压力。
- **指令被 Claude 偷懒跳过了**：说明你的命令“权重”不够。在 Markdown 里，越靠前的指令越容易被重视；加上 `CRITICAL:` 或者 `## 必须遵守` 等强烈词汇，甚至可以用 Python 脚本代替单纯的 Markdown 话术来执行关键校验。

## 分发与团队共享

你写出了牛逼的 Skill，独乐乐不如众乐乐。目前最推荐的分发模式：

**项目内即插即用（个人/小团队）：**
直接把 `.claude/skills` 整个文件夹通过 Git 推送到代码库。团队新人拉取代码后，什么都不用配，敲击 `/你的命令` 即可享受这套 SOP。

**拥抱开源（社区分发）：**
1. 在 GitHub 上新建公开仓库托管你的 Skill（此时，请在你的代码仓库**根目录**提供给人类看的 `README.md` 和演示截图）。
2. 在 MCP 服务器说明文档中，增加一个专门的 `[Your Service] Skill 安装指南` 节点，解释：“MCP 提供了连接能力，而这个 Skill 教会了 Claude 如何完美运用这种能力。”

## 总结：你的 AI 不再是一个单纯的回答者

通过这 5 大模式，你会发现，我们正在把 Claude Code 从一个 **“你问它答的对话框”**，变成了一个具备**流程控制、条件分支、自我监控和专业知识约束**的微型自动化程序。

现在，去你的代码库里挑选一个最让你头疼的工作流，用上面的一种模式，给你的 Claude 写一份《SOP 操作手册》吧！

## 🎁 附录：Skill 上线前防翻车自查清单 (Cheat Sheet)

无论是个人使用还是分享给团队，提交前请务必核对这 8 条红线：

- [ ] 文件夹名称和 `name` 字段是否完全一致且为 kebab-case（如 `my-tool`）？
- [ ] 核心文件是否精确命名为 `SKILL.md`（不能有空格，全大写）？
- [ ] YAML 头部是否被 `---` 完好包裹，且绝对没有使用 `< >` XML 标签？
- [ ] `description` 里是否明确写出了**触发器**（“当用户说 xxx 时使用”）？
- [ ] 测试过“正常请求”下是否会 100% 触发？
- [ ] 测试过“无关请求”下是否会被静默忽略（未过度触发）？
- [ ] 如果使用了 MCP，是否测试过 API 报错时的兜底话术？
- [ ] 代码库中，是不是把 README 写在了外层，而不是塞进了 `skills/` 文件夹里？
